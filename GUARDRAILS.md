# GUARDRAILS.md — Meeting Scheduler

Жёсткие ограничения для AI-агента. При конфликте между "как удобнее реализовать" и этим файлом — всегда побеждает этот файл.

## Запрещено

1. **Не менять зафиксированный скоуп ТЗ.** Реализуем ровно 3 пункта: создание встречи, просмотр расписания (день/неделя), запрет пересечения слотов. Не добавлять регистрацию пользователей, авторизацию, уведомления и т.п. без явного запроса — это тестовое задание, а не продукт, лишний функционал размывает оценку.
2. **Не хранить секреты в коде.** Любые ключи (API-ключи LLM для уровня B и т.п.) — только через `.env`, файл `.env` должен быть в `.gitignore`. В репозиторий коммитить только `.env.example`.
3. **Не удалять и не переписывать тесты, чтобы они "проходили".** Если тест падает — чинить логику, а не тест, если только сам тест не содержит явную ошибку (и это нужно явно обозначить в отчёте).
4. **Не занижать проверку конфликтов "для простоты".** Проверка пересечения интервалов должна учитывать частичное пересечение, полное вложение и разных участников — упрощение здесь = проваленный основной критерий ТЗ.
5. **Не смешивать уровень A и уровень B (AI-слой).** `ai_layer/` не должен быть зависимостью для основного API — если LLM/сеть недоступны, базовый функционал (создание/просмотр встреч, проверка конфликтов) обязан работать полностью автономно.
6. **Не молчать о неопределённости.** Любое неочевидное архитектурное решение (TZ, формат недели, поведение при пустом списке участников) — фиксировать в README как явное допущение, а не решать втихую и не спрашивать бесконечно — см. AGENTS.md, раздел "Что делать при неопределённости".
7. **Не коммитить закомментированный код и мёртвые ветки логики.**
8. **Не использовать SQLite-специфичные хаки**, которые сломаются при переходе на Postgres (например, полагаться на конкретный формат хранения дат SQLite) — писать через SQLAlchemy ORM/Core, не через сырые строковые сравнения дат.

## Обязательно

1. Каждая новая бизнес-логика — с тестом до или сразу после реализации, не "потом".
2. Время в БД — только UTC.
3. Ответ API при конфликте слота — `409`, с понятным телом ответа (какой участник, какая встреча пересекается).
4. Перед тем как считать этап законченным — прогнать `pytest` и линтер, приложить результат в отчёте.
5. README должен позволять человеку, не участвовавшему в разработке, поднять проект с нуля за 5 минут.

Если в процессе работы обнаруживается противоречие между PLAN.md и GUARDRAILS.md, или требование ТЗ невозможно реализовать в рамках стека без нарушения guardrails — остановиться и явно сообщить об этом Андрею, а не принимать решение молча.
