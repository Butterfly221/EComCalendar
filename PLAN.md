# PLAN.md — Meeting Scheduler (тестовое задание AI Developer)

## 1. Цель

Сервис для сотрудников компании:
1. Сотрудник может добавить встречу: кто с кем встречается и когда.
2. Любой сотрудник может посмотреть расписание (день/неделя): кто с кем и когда встречается.
3. Система не позволяет занять слот, если у кого-то из участников уже есть встреча в это время.

## 2. Уровни реализации

**Уровень A (обязательный, MVP)** — закрывает ТЗ буквально: REST API + хранилище + проверка конфликтов + тесты.

**Уровень B (опциональный, AI-слой)** — LLM-агент с function calling поверх API: сотрудник пишет "Запланируй встречу с Аней завтра в 15:00 на 30 минут" — агент сам вызывает нужные tools (create_meeting, get_schedule) и обрабатывает конфликт, предлагая альтернативные слоты. Это демонстрирует именно AI Developer скиллы, а не просто backend.

Рекомендация: сделать A полностью и качественно (это и есть критерий приёмки), B — как "бонус" в отдельной ветке/папке, если останется время. Не размазывать усилия — А должен работать безупречно.

## 3. Стек

- Python 3.11+, FastAPI, Pydantic v2
- SQLite + SQLAlchemy (async) — для теста достаточно, в README указать путь миграции на Postgres
- pytest + httpx (для интеграционных тестов API)
- Уровень B: LangGraph, тонкая обёртка, без RAG (тут не нужен)

## 4. Модель данных

```
Employee
  id, name, email

Meeting
  id, title, start_time (UTC), end_time (UTC), created_by (Employee.id)

MeetingParticipant (many-to-many)
  meeting_id, employee_id
```

Важно: время хранить в UTC, конвертацию в локальную MSK делать на выводе.

## 5. Этапы разработки

1. **Инициализация проекта** — структура папок, requirements.txt, .env.example, AGENTS.md, GUARDRAILS.md, PLAN.md, README.md (скелет).
2. **Модели и слой хранения** — SQLAlchemy models, Pydantic schemas (create/read), Alembic (опционально, можно и `Base.metadata.create_all` для теста).
3. **Бизнес-логика конфликтов** — функция `has_conflict(participant_ids, start, end, exclude_meeting_id=None) -> bool`.
   Условие пересечения интервалов: `existing.start_time < new.end_time AND existing.end_time > new.start_time`.
   Важно НЕ считать конфликтом встречи впритык (10:00–10:30 и 10:30–11:00).
4. **Эндпоинты FastAPI**:
   - `POST /meetings` — создать встречу. Валидация участников, `start < end`, проверка конфликтов → 409 Conflict с указанием, у кого именно пересечение.
   - `GET /meetings?date=YYYY-MM-DD` — список встреч на день.
   - `GET /meetings?week_start=YYYY-MM-DD` — список встреч на неделю.
   - `GET /employees/{id}/meetings` — расписание конкретного сотрудника (опционально, но полезно).
5. **Тесты (обязательно)**:
   - Юнит-тесты на `has_conflict`: полное пересечение, частичное пересечение, встык (не конфликт), разные участники (не конфликт), один и тот же участник в двух встречах.
   - Интеграционные тесты API: создание, конфликт → 409, выборка по дню/неделе.
6. **Гонки при параллельных запросах** — упомянуть в README как известное ограничение MVP; при необходимости — уникальный констрейнт/транзакция с блокировкой на уровне БД.
7. **README.md** — как запустить (`uvicorn app.main:app`), примеры curl/HTTPie запросов, схема данных, известные ограничения.
8. **(Опционально) Docker-compose** — plus к тестовому, показывает production-mindset.
9. **(Опционально, уровень B) AI-слой** — отдельная папка `ai_layer/`, не трогает ядро API, вызывает его как обычный HTTP-клиент. Явно описать в README, что это демонстрационный бонус, а не замена основного функционала.
