# AGENTS.md — Meeting Scheduler

Документ для AI-агента (DeepSeek Coder / Continue.dev), работающего над этим проектом. Читать перед началом каждой сессии.

## Контекст проекта

Тестовое задание на позицию AI Developer (Fashion e-commerce). Сервис бронирования встреч между сотрудниками с проверкой пересечений слотов. Полное описание требований — в PLAN.md. Ограничения — в GUARDRAILS.md, они имеют приоритет над любыми удобными решениями "чтобы было проще".

## Роль агента

Ты выступаешь как парный разработчик под руководством Андрея. Твоя задача — реализовывать этапы из PLAN.md по одному, а не пытаться сгенерировать весь проект одним махом. После каждого этапа — короткий отчёт: что сделано, что осталось, какие тесты добавлены.

## Структура проекта (целевая)

```
app/
  main.py            # FastAPI app, роуты
  models.py          # SQLAlchemy модели
  schemas.py         # Pydantic схемы
  crud.py            # логика работы с БД
  conflicts.py       # логика проверки пересечений слотов
  database.py        # engine, session
tests/
  test_conflicts.py
  test_api.py
ai_layer/            # опционально, уровень B — не трогать до завершения app/
README.md
PLAN.md
AGENTS.md
GUARDRAILS.md
requirements.txt
.env.example
```

## Рабочий процесс

1. Перед изменением кода — свериться с PLAN.md, к какому этапу относится задача.
2. Изменения делать маленькими шагами: один этап плана = один логический коммит.
3. Для любой новой бизнес-логики (особенно `conflicts.py`) — сначала написать тест, потом реализацию.
4. Не расширять API сверх того, что описано в PLAN.md, без явного запроса Андрея (см. GUARDRAILS.md).
5. По завершении этапа — прогнать `pytest` и `ruff check .`, отчитаться о результате.

## Соглашения по коду

- Python 3.11+, type hints обязательны на публичных функциях.
- Время — только UTC внутри системы, конвертация в MSK — на границе (API response).
- Неделю считать с понедельника.
- FastAPI роуты — тонкие, вся логика — в `crud.py` / `conflicts.py`.
- Ошибки конфликта возвращать как `409 Conflict` с телом, объясняющим, у какого участника пересечение (это важно для UX и для проверяющего — показывает внимание к деталям).
- Комментарии — только там, где логика неочевидна (например, формула пересечения интервалов). Не комментировать очевидное.

## Definition of Done для агента

Этап считается завершённым, когда:
- код написан и покрыт тестами,
- `pytest` проходит,
- `ruff`/`black` без замечаний,
- в отчёте явно указано, какой пункт PLAN.md закрыт.

## Что делать при неопределённости

Если требование неоднозначно — не выдумывать молча. Зафиксировать явное допущение в README.md в разделе "Известные допущения" и продолжить работу с этим допущением. Не останавливать весь процесс ради одного уточнения.
